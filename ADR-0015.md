# ADR-0015: DamageReport Creation Lifecycle

## Status
Approved

## Context
In the previous implementation, Damage Reports and their Assessment Items were often created in a single batch, or items were allowed to be added before the report itself was formally established in the system. This led to orphaned items and numbering inconsistencies.

## Decision
The creation of a `DamageReport` is now a strictly ordered two-phase workflow:

### Phase 1: Header Creation
1. The user selects a Farm.
2. The user enters the Damage Date, Agricultural Sector, and Cause.
3. The user saves the Header.
4. The system performs a duplicate check (`FarmId + DamageDate`).
5. If unique, a `DamageReport` is created and an official `ReportNumber` is generated immediately.
6. The initial status is set to `PendingTechnicalVerification`.

### Phase 2: Damage Assessment
1. Only after the Header is saved and an identity is established, the user can proceed to add Damage Items.
2. Items inherit the `Geographic Scope` from the Header (resolved via the `Farm` entity).
3. The user selects the `Damage Nature` and `Damage Action` specifically for each assessment item.

## Rules
- Users cannot move to the Assessment phase without a successfully saved Header.
- Re-opening an existing report for the same Farm/Date will load the established Header and its items.
- Nature selection is locked after the first assessment item is added to maintain categorization consistency.
- **Entity Principle**: `DamageReport` only stores the incident header and state. Ownership and geographic context (Governorate, Directorate, etc.) are fetched from the associated `Farm` record to maintain a single source of truth.

## Consequences
- Reduces data entry errors by establishing context first.
- Prevents duplicate reports for the same event.
- Aligns the digital system with the physical form-filling process used by field engineers.
