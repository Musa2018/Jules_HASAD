# ADR-0016: Damage Reference Data Ownership & Management

## Status
Approved

## Context
Damage Reference Data (Natures, Actions, Categories, etc.) was previously managed using EF Core `HasData` in the `ApplicationDbContext`. While this ensures data is present, it becomes problematic during schema refactoring (renames, column shifts) as EF Core generates `InsertData` operations in multiple migrations. If a database already contains these rows (seeded by a previous version or `DbInitializer`), the migration fails with duplicate primary key violations.

Additionally, multi-instance deployments could trigger concurrent seeding operations, leading to race conditions.

## Decision
We have established a "Single Owner" architecture for Damage Reference Data:

1. **Owner**: `DbInitializer.cs` is the authoritative and single owner of Damage Reference Data.
2. **Migration Role**: EF Core migrations are strictly restricted to **Schema Management** only (Tables, Columns, Indexes, Relationships). All `InsertData` operations for reference lookups have been removed from migration files.
3. **Concurrency Safety**: `DbInitializer.SeedDamageReferenceDataAsync` uses a database-level application lock (`sp_getapplock`) to ensure only one instance performs the seeding at a time.
4. **Idempotency**: Seeding logic is fully idempotent. It performs per-record checks and `UPSERT` behavior (or `INSERT` skip) to preserve existing rows and maintain stable Lookup IDs.

## Rules
- Reference data must NOT be defined in `OnModelCreating` via `HasData` for tables owned by `DbInitializer`.
- Migrations must be audited to remove any auto-generated `InsertData` or `DeleteData` blocks for these tables.
- All new reference rows must be added to the `DbInitializer.cs` dataset.

## Consequences
- **Safety**: Existing databases can be upgraded to new schemas without data loss or primary key conflicts.
- **Stability**: Lookup IDs remain consistent across environments, which is critical for offline synchronization on mobile clients.
- **Portability**: Fresh database installations remain consistent as `DbInitializer` will seed the exact same dataset after schema creation.
- **Complexity**: Requires manual auditing of migrations when entities are modified, as EF Core might attempt to re-generate seed operations.
