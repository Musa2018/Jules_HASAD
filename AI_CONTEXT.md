# HASAD - AI Context Source

This document provides persistent context for AI agents working on the HASAD (Agricultural Damage Assessment System) project. It outlines the project's purpose, technical stack, architecture, and current development status.

## 1. Project Overview
- **Purpose**: A comprehensive solution for documenting and assessing agricultural damage in Palestine.
- **Backend Stack**:
  - .NET 8 Web API
  - Clean Architecture (Domain, Application, Infrastructure, API)
  - CQRS with MediatR
  - EF Core with SQL Server
  - ASP.NET Identity with JWT Authentication & Refresh Tokens
- **Flutter Stack**:
  - Riverpod (State Management)
  - Drift (Offline SQLite Database)
  - Dio (Networking with Auth Interceptors)
  - Freezed & JSON Serializable (Data Models)
  - GoRouter (Navigation)
  - Localization (Arabic/English)

## 2. Architecture
- **Clean Architecture Rules**:
  - Domain layer is the core, containing entities and interfaces.
  - Application layer contains CQRS commands/queries and business logic.
  - Infrastructure handles persistence and external services.
  - API handles HTTP requests and routing.
- **Offline-First Strategy**:
  - **Local First**: All data MUST be saved locally to Drift before attempting API calls for core features.
  - **Sync Queue**: Operations are recorded in `SyncQueue` and processed by `BackgroundSyncService`.
  - **Idempotency**: All creation commands use `ClientId` (UUID generated by mobile) to prevent duplicates.
  - **Concurrency**: Optimistic concurrency via `RowVersion` (byte array/timestamp).
- **User Scoping**:
  - Three levels: Global, Governorate, Directorate.
  - Regional isolation enforced via geographic IDs in session and commands.

## 6. Geographic Architecture
- **Entities**:
  - `Governorate`: Main administrative regions (16 Palestinian governorates).
  - `Locality`: Cities and villages linked to a governorate (Governorate 1 -> * Localities).
- **API**:
  - `GET /api/v1/Location/governorates`: Fetch active governorates.
  - `GET /api/v1/Location/localities?governorateId={id}`: Filtered locality lookup.
- **Mobile**:
  - Shared `location` feature providers (`governoratesProvider`, `localitiesProvider(govId)`).
  - Used by both Admin and Farmers modules.

## 7. Offline Architecture Status
- **Farmer Data**: Strictly **Offline-First**. All writes (Create/Update) happen in Drift first, then added to `SyncQueue`.
- **Location Data**: Treated as **Reference Data**. Currently retrieved online via shared providers to ensure data accuracy.
- **Sync Infrastructure**: `SyncQueue` and `BackgroundSyncService` remain the standard mechanism for data eventual consistency.
